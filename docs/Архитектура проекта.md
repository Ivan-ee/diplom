## 4. Взаимодействие Frontend ↔ Backend

### 4.1. Схема коммуникации

```
Next.js (Frontend, :3000)              NestJS (Backend, :4000)
┌─────────────────────┐                ┌─────────────────────┐
│                     │                │                     │
│  Server Components ─┼── fetch() ──→ │  GET /api/products  │
│  (SSR/SSG)          │                │  GET /api/products/:slug │
│                     │                │                     │
│  Client Components ─┼── fetch() ──→ │  POST /api/auth/*   │
│  (useEffect, SWR)   │                │  POST /api/orders   │
│                     │                │  GET /api/constructor/ingredients │
│  Zustand stores ────┼── fetch() ──→ │  POST /api/upload   │
│                     │                │  PUT /api/admin/*   │
│                     │                │                     │
└─────────────────────┘                └─────────────────────┘
        ↕                                      ↕
  Статические ассеты                    Drizzle ORM → PostgreSQL
  (models/, textures/)                  MinIO (S3)
```

### 4.3. Proxy в Next.js

Next.js проксирует API-запросы на NestJS через `next.config.ts`:

```typescript
// next.config.ts
const nextConfig = {
  async rewrites() {
    return [
      {
        source: '/api/:path*',
        destination: 'http://api:4000/api/:path*',
      },
    ];
  },
};
```

Для Server Components — прямой fetch на внутренний адрес Docker-сети:

```typescript
// В Server Component
const products = await fetch('http://api:4000/api/products', {
  next: { revalidate: 3600 }
}).then(r => r.json());
```

## 6. Ключевые архитектурные решения

### 6.1. Рендеринг страниц (Next.js)

|Страница|Стратегия|Обоснование|
|---|---|---|
|Главная|SSG + ISR (revalidate: 3600)|Контент меняется редко, SEO важен|
|Каталог|SSR|Фильтры в URL, SEO для карточек|
|Карточка товара|SSG + ISR|Генерация при первом запросе, кеш|
|Конструктор|CSR (Client Component)|Тяжёлый 3D, интерактивность|
|Корзина / Checkout|CSR|Персональные данные, без SEO|
|ЛК|CSR|Защищённые данные|
|Админ-панель|CSR|Только для авторизованных|

### 6.3. Потоки данных

**Поток «Заказ готового торта»:**

```
Каталог → Карточка (выбор веса) → Корзина → Checkout (самовывоз) → POST /api/orders → БД
```

**Поток «Заказ через конструктор»:**

```
Конструктор (Zustand: constructor-store)
  → Пользователь собирает торт (шаги 1-5)
  → Калькулятор пересчитывает цену (client-side)
  → «Добавить в корзину»
  → gl.toDataURL → скриншот → POST /api/upload → MinIO
  → CakeConfig JSON + screenshotUrl → cart-store
  → Checkout (самовывоз) → POST /api/orders → БД
```

**Поток «Админ просматривает заказ»:**

```
Админ → GET /api/admin/orders → Список заказов → Детали
  → Если тип = constructor: показать скриншот + распарсить CakeConfig
  → PUT /api/admin/orders/:id/status → Смена статуса
```

### 6.4. Авторизация и роли

Две роли: `USER` (по умолчанию) и `ADMIN`.

**NestJS сторона (основная):**

- Passport.js + JWT Strategy
- `JwtAuthGuard` — проверяет токен на защищённых эндпоинтах
- `RolesGuard` + `@Roles('ADMIN')` декоратор — ограничивает доступ к админским эндпоинтам
- `POST /api/auth/register` → создаёт пользователя, возвращает JWT
- `POST /api/auth/login` → проверяет credentials, возвращает JWT

**Next.js сторона:**

- Middleware проверяет JWT (из cookie) для защиты клиентских роутов (`/account/*`, `/admin/*`)
- Хранение токена: httpOnly cookie (устанавливается бэкендом)

### 6.5. NestJS — модули и зависимости

```
AppModule
├── AuthModule          (Passport, JWT, UsersService)
├── ProductsModule      (CRUD товаров, фильтрация)
├── ConstructorModule   (Ингредиенты, бисквиты, начинки, декор, цены)
├── OrdersModule        (Создание, история, смена статуса)
├── UploadModule        (MinIO presigned URLs, загрузка файлов)
├── AdminModule         (Дашборд, статистика, управление)
└── DatabaseModule      (Drizzle provider, подключение к PostgreSQL)
```

### 6.6. Загрузка и хранение файлов

Все изображения хранятся в MinIO (S3-совместимое хранилище):

|Бакет|Содержимое|Доступ|
|---|---|---|
|`products`|Фото готовых товаров|public read|
|`textures`|Текстуры для 3D-конструктора|public read|
|`screenshots`|Скриншоты собранных тортов|private|
|`models`|3D-модели (.glb)|public read|

Загрузка через presigned URL:

```
Клиент → POST /api/upload (получает presigned URL от NestJS) → PUT файл в MinIO
```

## 9. Порядок разработки

| Этап                               | Что делаем                                                 | Зависимости |
| ---------------------------------- | ---------------------------------------------------------- | ----------- |
| **1. Инфраструктура**              | Turborepo, Docker Compose (postgres + minio), конфиги, ENV | —           |
| **2. База данных**                 | packages/db: Drizzle-схемы, миграции, seed                 | Этап 1      |
| **3. NestJS scaffold**             | apps/api: модули, Swagger, DatabaseModule, JWT             | Этап 2      |
| **4. 3D-Конструктор**              | apps/web: R3F-сцена, модели, материалы, шаги, анимации     | Этап 1      |
| **5. API: Products + Constructor** | NestJS: CRUD товаров, ингредиенты, калькулятор             | Этап 3      |
| **6. Каталог + Landing**           | Next.js: SSR-страницы, карточки, фильтры                   | Этап 5      |
| **7. API: Auth + Orders**          | NestJS: JWT, регистрация, заказы                           | Этап 3      |
| **8. Корзина + Checkout**          | Next.js: Zustand, корзина, оформление (самовывоз)          | Этап 4, 7   |
| **9. ЛК + Админ**                  | Next.js: история заказов, админ-панель                     | Этап 7      |
| **10. Полировка**                  | Анимации конструктора, FPS, мобильная адаптация, тесты     | Все         |

> **Этапы 3 и 4 запускаются параллельно:** NestJS API и 3D-конструктор не зависят друг от друга на начальном этапе. Конструктор работает с mock-данными, пока API не готово.

## 10. Нефункциональные требования

- **3D-производительность:** 60 FPS на десктопе, 30+ на мобильном. Low-poly модели (< 50k полигонов), текстуры ≤ 1024×1024, lazy loading ассетов.
- **Lighthouse:** каталог — Performance > 80, LCP < 2.5s.
- **Изображения:** WebP/AVIF через next/image, lazy loading.
- **Бандл 3D:** динамический импорт R3F (`next/dynamic` с `ssr: false`).
- **API:** Swagger-документация, DTO-валидация, единый формат ответов.
- **Доступность:** fallback без WebGL — 2D-превью для слабых устройств.
